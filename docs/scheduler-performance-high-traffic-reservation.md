# 当前调度架构在高并发预约场景下的优势与边界

## 结论先说

这套实现的优势不在于“用了 Redis”，而在于把高并发预约场景里最容易成为瓶颈的几个点拆开了：

1. 写请求先进入 `mq`，不会直接压到 scheduler HTTP 线程
2. 近未来任务先预热到 Redis，避免高峰期持续扫 MySQL
3. 任务按 shard 分散到多个 scheduler 实例处理
4. worker 选点看实时负载，而不是简单随机
5. 冷数据从在线热表里迁走，避免热表越来越大

所以它更像一个“面向高并发预约场景做过专项优化的调度内核”，而不是一个开箱即用的通用任务平台。

## 为什么这个场景难

高并发预约类场景通常有几个特点：

- 短时间内创建大量任务
- 大量任务集中在某几个时刻到期
- 调度结果会直接影响业务资源占用
- 执行失败、超时、回调失败都需要兜底

如果还是采用“客户端直接打调度器，调度器自己扫库、自己选点、自己同步扛住峰值”的方式，瓶颈会很快集中到 scheduler 自身。

## 当前架构为什么更适合

### 1. `mq` 把写入峰值和调度执行峰值拆开

客户端创建任务、worker 回调完成结果，现在都先写入 Redis Stream。

好处：

- scheduler 不需要直接承担外部写流量
- 调度核心可以按自己的消费速度处理命令
- 高峰写入和实际调度执行解耦

代价：

- 写接口变成“已接收”语义，而不是“已完成”语义
- 当前实现返回的是 `streamRecordId`，不是同步结果

### 2. 近未来预热降低了 MySQL 热路径压力

当前实现不是让 scheduler 在高峰期持续扫 `sj_job`，而是：

1. 远未来任务留在 MySQL
2. 近未来任务由 `preheater` 预热到 Redis
3. scheduler 主要操作 Redis `ready/inflight/detail`

这样高峰期真正频繁访问的是 Redis 热路径，而不是 MySQL 范围扫描。

### 3. 多 scheduler 通过 shard lease 并行工作

多个 scheduler 会：

1. 注册成员
2. 基于 Redis lease 获得 shard 所有权
3. 只处理自己持有的 shard

这比“所有节点抢一个共享队列”更容易横向扩展，也更容易定位热点。

### 4. worker 选点不是盲派

worker 持续把 `currentLoad/maxLoad/heartbeat` 写入 Redis。

scheduler 通过 Lua 批量 reserve 容量，而不是每个任务都做一次独立选点。

这意味着：

- 调度更容易避开已满节点
- Redis 往返次数更少
- 集中到期时的选点成本更可控

### 5. 冷热分表避免在线链路被历史数据拖慢

在线调度只关心热表 `sj_job`，归档后任务进 `sj_job_archive`。

这样能减少：

- 热表扫描范围
- 热表索引膨胀
- 在线 dashboard 查询干扰

## 和通用方案相比，优势在哪里

如果和典型的“中心调度 + 直连写入”方案相比，当前实现更偏向下面这些目标：

- 更强的突发写入承压能力
- 更低的近未来任务调度延迟
- 更好的多 scheduler 并行能力
- 更容易把执行节点实时负载纳入调度决策

但它不是在所有场景都更优。

## 真实瓶颈会在哪里

这套架构并不自动等于“无限扩展”，真实瓶颈通常在这些位置：

### 1. shard 分布是否均匀

如果内部 `shardKey` 的生成策略设计得不好，热点还是会压到少数 shard 上。

### 2. Redis 热 key / 热 slot

即使做了 shard，某些 `ready/inflight/detail` key 仍可能特别热。

### 3. scheduler 到 worker 的分发链路

当前仍是 scheduler 主动调用 worker HTTP 接口。

如果任务爆发太猛，瓶颈可能转移到：

- scheduler 分发线程池
- HTTP 连接数
- worker host 网络抖动

### 4. worker 自身执行能力

最终任务还是在 worker 上执行。

如果 worker 业务逻辑很重，调度再快也只是把瓶颈更早地暴露出来。

### 5. MQ 写接口的异步语义

现在 `mq` 只保证“请求已入队”，不保证“业务已处理完成”。

这对削峰是好事，但对调用方意味着：

- 需要接受异步结果语义
- 如果要强一致回执，还需要补命令结果查询或回传机制

## 当前实现的边界和残留风险

1. Redis Stream 消费是至少一次语义，不是严格 exactly-once
2. 当前没有额外的消费去重层，业务执行后、ACK 前如果进程异常，理论上可能重复消费
3. `mq` 目前没有单独的命令结果查询接口
4. `worker` 仍执行模拟任务，不是业务真实逻辑

## 适合什么，不适合什么

更适合：

- 高并发预约
- 近未来任务密集触发
- 调度和执行都需要横向扩展
- 更关心吞吐和调度延迟，而不是平台化运营能力

不一定适合：

- 普通定时任务场景
- 更看重控制台、权限、审计、运营能力的场景
- 强依赖同步创建结果回执的调用模式
